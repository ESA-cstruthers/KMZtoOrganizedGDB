KMZ to Organize GDB — Decision Flow

Walk the waterfall; see what comes out KMZ to Organize GDB (Waterfall)
Every output Feature Class includes: Name, FolderPath, SourceKMZ, RawPopup, plus all parsed popup attributes as typed fields. Folder1/Folder2/Folder3 columns are added whenever folder context isn't already in the FC or FD name. Layout uses only the last 3 folder segments. Deeper context stays in the FolderPath field.
Sample input: ProjectA.kmz Birds 2026 / BUOW / Known (Pt)  +  Birds 2026 / BUOW / Suspected (Pt) | ProjectB.kmz Wildlife / Coyote / Den (Pt)  +  Wildlife / Coyote / Trail (Line)
The waterfall
Input KMZ / KML files
1 Decision
Output container?
Merged: 1 GDB with the FD/FC layout from below.
Per-KMZ: N GDBs (one per source, schema per source); same layout inside each.
2 Decision
Split output into Feature Datasets?
3a Process
Feature Dataset source
Top folder (segments[-3]) — e.g. "Birds_2026"
Top two folders (segments[-3] + [-2]) — e.g. "Birds_2026_BUOW"
Source KMZ filename — "ProjectA" (merged container only)
3b Process (FDs on)
Feature Class naming
Per-geometry only — Points / Polylines / Polygons
Leaf folder name — "Known", "Suspected"   (only when 3a = Top two folders)
Leaf + geometry — e.g. "Known_Points"
Parent + leaf + geometry — e.g. "BUOW_Known_Points"
Folder pair + geometry — e.g. "Birds_2026_BUOW_Points"
3b Process (FDs off)
Feature Class naming (at GDB root)
Per-geometry only — Points / Polylines / Polygons
Top folder + geometry — e.g. "Birds_2026_Points"
Leaf + geometry — e.g. "Known_Points"
Parent + leaf + geometry — e.g. "BUOW_Known_Points"
Folder pair + geometry — e.g. "Birds_2026_BUOW_Points"
Example combinations
Folder Hierarchy (merged)
1. One merged GDB   2. FDs Yes   3a. Top two folders   3b. Leaf folder name
Output.gdb + Birds_2026_BUOW/ - Known - Suspected + Wildlife_Coyote/ - Den - Trail
When source folder structure already matches the deliverable.
Source-tracked + Folder-pair FCs (merged)
1. One merged GDB   2. FDs Yes   3a. Source KMZ   3b. Folder pair + geometry
Output.gdb + ProjectA/ - Birds_2026_BUOW_Points + ProjectB/ - Wildlife_Coyote_Points - Wildlife_Coyote_Polylines
When KMZ origin matters AND folder context belongs in FC names.
Per-KMZ + folder-pair FCs (NEW)
1. One GDB per KMZ   2. FDs No   3b. Folder pair + geometry
ProjectA.gdb - Birds_2026_BUOW_Points ProjectB.gdb - Wildlife_Coyote_Points - Wildlife_Coyote_Polylines
Each input KMZ becomes a separate deliverable with named FCs.
Per-KMZ flat (simplest split)
1. One GDB per KMZ   2. FDs No   3b. Per-geometry only
ProjectA.gdb - Points ProjectB.gdb - Points - Polylines
Each input KMZ separate, generic per-geometry FCs inside.
